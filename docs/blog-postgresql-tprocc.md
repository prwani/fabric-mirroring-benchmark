# Benchmarking Microsoft Fabric Mirroring with Azure Database for PostgreSQL and HammerDB TPROC-C

Microsoft Fabric Mirroring is most useful when it keeps operational data fresh for analytics. For that reason this benchmark uses HammerDB **TPROC-C**, an OLTP workload, instead of an analytical TPROC-H/TPC-H dataset.

This post covers the PostgreSQL run in the series. Azure SQL Database has its own post in [`blog-azure-sql-db-tprocc.md`](blog-azure-sql-db-tprocc.md).

## What we built

```text
HammerDB VM
  -> Azure Database for PostgreSQL Flexible Server
      -> Microsoft Fabric Mirroring
          -> Fabric mirrored database SQL endpoint
              -> Row-count and CDC latency measurement
```

The environment includes:

- Azure Database for PostgreSQL Flexible Server.
- Linux benchmark VM with HammerDB 5.0 and PostgreSQL client tools.
- Microsoft Fabric capacity and workspace.
- TPROC-C schema/data generated by HammerDB.
- `fabric_cdc_latency_marker` for controlled CDC latency checks.

## Prerequisites

| Requirement | Notes |
|---|---|
| Azure subscription | Permission to deploy the source, VM, networking, monitoring, and Fabric capacity. |
| Microsoft Fabric tenant | Permission to create a workspace and mirrored database item. |
| PostgreSQL Flexible Server | General Purpose or Memory Optimized tier for mirroring support. |
| Benchmark VM access | SSH or Azure VM Run Command. |
| Fabric SQL endpoint access | Used for row-count and marker visibility checks. |

Validated defaults:

|---|---|

| Scale | 10 warehouses |

| Timed run | 8 VUs, 2-minute ramp-up, 10-minute duration |
| Fabric capacity | F8 |

## Steps

1. Deploy the Azure baseline:

   ```bash
   SOURCE_TYPE=postgresql scripts/provision/deploy-azure.sh
   ```

2. Prepare PostgreSQL mirroring prerequisites:

   ```bash
   export PGPASSWORD="$POSTGRES_ADMIN_PASSWORD"

   psql "host=$POSTGRES_HOST port=5432 dbname=$POSTGRES_DATABASE user=$POSTGRES_ADMIN_USER sslmode=require" \
     -v ON_ERROR_STOP=1 \
     -f scripts/provision/setup-postgres-mirroring-prereqs.sql
   ```

3. Install HammerDB on the benchmark VM:

   ```bash
   scripts/provision/install-hammerdb.sh
   ```

4. Build TPROC-C:

   ```bash
   export TPROC_C_DATABASE=tprocc
   export TPROC_C_USER=tprocc
   export TPROC_C_PASSWORD="<benchmark-user-password>"
   export TPROC_C_WAREHOUSES=10
   export TPROC_C_BUILD_VUSERS=4

   "${HAMMERDB_CLI:-hammerdbcli}" auto scripts/benchmark/hammerdb-build-tprocc.tcl
   "${HAMMERDB_CLI:-hammerdbcli}" auto scripts/benchmark/hammerdb-check-tprocc.tcl
   ```

5. Create the marker table after the HammerDB build:

   ```bash
   psql "host=$POSTGRES_HOST port=5432 dbname=$TPROC_C_DATABASE user=$POSTGRES_ADMIN_USER sslmode=require" \
     -v ON_ERROR_STOP=1 \
     -f scripts/provision/setup-tprocc-marker.sql
   ```

6. Create the Fabric mirrored database item for PostgreSQL and select:

   | Table |
   |---|
   | `warehouse` |
   | `district` |
   | `customer` |
   | `item` |
   | `stock` |
   | `orders` |
   | `new_order` |
   | `order_line` |
   | `history` |
   | `fabric_cdc_latency_marker` |

7. Validate initial sync by comparing source and Fabric row counts.

8. Run the HammerDB timed workload and marker latency tests.

## Notes

- Use a dedicated TPROC-C benchmark role; HammerDB creates/alters the configured role during schema build.
- Create the marker table only after HammerDB finishes building the schema, because HammerDB expects an empty target database.
- Use marker-table latency as the benchmark source of truth. Fabric status APIs are useful context, but marker visibility measures user-observable freshness.
- Keep PostgreSQL and Azure SQL results in separate posts so the series can add more source systems without rewriting earlier results.
